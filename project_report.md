# Отчет по проекту LearnHub (Project Report)

Настоящий отчет предназначен для будущих программистов, присоединяющихся к разработке образовательной платформы **LearnHub**. Здесь описана ключевая архитектура, принятые технические решения, а также рекомендации по дальнейшему развитию.

---

## 1. Архитектура и Инфраструктура

Проект построен как **Monorepo** на базе **Turborepo** (`pnpm workspaces`). Это позволяет эффективно разделять код на пакеты, переиспользовать интерфейсы и типы между фронтендом и бэкендом, а также кэшировать процессы сборки.

### 1.1. Фронтенд (`apps/web`)
- **Фреймворк**: Next.js 15+ (App Router).
- **Стилизация**: Tailwind CSS (с настроенной дизайн-системой Ocean-Blue в `globals.css`).
- **PWA**: Интегрирован `@ducanh2912/next-pwa`. Приложение может работать автономно на мобильных устройствах, имеет свой `manifest.json` и Service Worker. Компонент `PwaInstallPrompt` отлавливает браузерные события для установки.
- **Особенности**:
  - В приложении используется строгая маршрутизация по ролям (`(admin)`, `(author)`, `(student)`, `(auth)`). 
  - Дизайн построен на паттернах **Glassmorphism** и интерактивных микроанимациях.

### 1.2. Бэкенд (`apps/api`)
- **Фреймворк**: NestJS (TypeScript).
- **Зоны ответственности**: Обработка API-запросов, проверка JWT-токенов авторизации, выдача информации о курсах и сохранение прогресса.
- **База данных**: PostgreSQL. Взаимодействие происходит через пакет `@repo/database`, который инкапсулирует Prisma Client.

### 1.3. Общие Пакеты (`packages/`)
- `@repo/database`: Хранит схемы (`schema.prisma`), скрипты миграций и инициализацию клиента Prisma.
- `@repo/shared`: Общие интерфейсы, типы, константы и утилиты, используемые как на сервере, так и на клиенте.
- `@repo/tsconfig`: Централизованные конфигурации TypeScript.

## 2. Ключевые Фичи и Механики

### 2.1. Авторизация и Роли
На платформе существует 3 базовые роли:
1. **Студент (`student`)**: Может проходить курсы, играть в мини-игры, зарабатывать очки в рейтингах.
2. **Автор/Преподаватель (`author`)**: Имеет панель для создания образовательного контента (курсы, квизы).
3. **Администратор (`admin`)**: Управляет всей платформой, создает уровни для мини-игр через визуальные конструкторы, модерирует контент.

### 2.2. Система Мини-игр (Геймификация)
Мини-игры встроены прямо в платформу (React компоненты) и являются клиентскими (Client-side logic), чтобы не нагружать бэкенд вычислениями физики и состояния:
- **Побег Робота (Robot Escape)**: 
  - *Суть*: Игрок составляет последовательность команд (вперед, направо, налево, атака), чтобы провести робота от старта к финишу.
  - *Технологии*: Движок на React состояниях (useState), спрайты, сетка (Grid).
  - *Особенности*: Редактор уровней позволяет админу расставлять ресурсы (аптечки, батареи), врагов (Зомби, Астероиды, Лава) и телепорты. Имеет систему тем ("Космос", "Зомби-апокалипсис", "Стандарт").
- **2D Майнкрафт (Minecraft Crafting)**:
  - *Суть*: Изучение функций и крафта. У студента есть инвентарь и верстак 3x3 (реализован через React DnD или аналогичный механизм перетаскивания). Игрок должен скрафтить инструмент, чтобы разрушить препятствие (Стив идет к препятствию и ломает его).
  - *Особенности*: Админ может задавать стартовый инвентарь и препятствия для каждого уровня. 

Уровни мини-игр могут храниться в базе данных (в формате JSON конфигураций) и загружаться динамически.

## 3. Деплой и DevOps

Развертывание (Deploy) платформы автоматизировано через набор Node.js скриптов-хелперов.
Сервер использует **PM2** для управления процессами Node.js и **Nginx** в качестве обратного прокси.

- **Сборка**: При вызове команды сборки (например, скрипт `update_and_rebuild_vps.js`), Turborepo компилирует все измененные пакеты.
- **PM2**: 
  - `learnhub-api` (порт 3001) - NestJS процесс.
  - `learnhub-web` (порт 3000) - Next.js процесс.
- **Скрипты**: Находятся в `scratch/deploy-helper/`. Они позволяют проверять статус процессов, смотреть логи, перезапускать приложение и обновлять SSL сертификаты (через certbot).

## 4. Рекомендации для будущих разработчиков

Если вы беретесь за развитие платформы, вот на что следует обратить внимание:

1. **Безопасность Мини-игр**: Сейчас валидация победы в мини-играх происходит на стороне клиента (браузера). Для защиты от читеров рекомендуется перенести верификацию команд (особенно в турнирах или рейтингах) на бэкенд. Клиент должен отправлять массив команд (например, `['forward', 'attack']`), а бэкенд — симулировать прохождение и возвращать результат.
2. **Оптимизация базы данных**: При большом росте количества студентов таблица `UserProgress` (отслеживающая прогресс по каждому уроку) начнет быстро расти. Рекомендуется использовать индексы по `userId` + `courseId`.
3. **Развитие PWA**: Текущая реализация поддерживает оффлайн-страницы (fallback). В будущем можно добавить **Push-уведомления** (Web Push Notifications), чтобы напоминать студентам о необходимости продолжить курс или о начале нового сезона рейтингов.
4. **Тестирование (QA)**: 
  - В проекте необходимо внедрить E2E тесты (на базе **Playwright** или Cypress) для тестирования сложных пользовательских сценариев (например, Drag-and-Drop в 2D Майнкрафте).
  - Добавьте Unit-тесты (Jest / Vitest) для проверки движков мини-игр (правильность вычисления позиций робота, обработка коллизий с врагами).
